# Оракул — это команда, а не файл

## Контекст
Phase 5 item 1 предписывал авторить `init.sh` каждому долгоживущему проекту: альтернатива — «or
documented one-liner» — пряталась в скобках посреди абзаца, под жирным «Runnable oracle + env
init», и де-факто дефолт читался как «сочини файл». Второй энтрипойнт, гоняющий те же гейты, —
источник дрейфа, а не оракул. Это рецидивная over-scaffolding ровно того класса, который
headline-принцип кита запрещает: kit нарушал его в собственной Phase 5.

Побочный эффект того же предписания — файл в корне репозитория: `init.sh` числился среди
«genuine product files», хотя это харнесная машинерия, а не продукт.

## Изменения
Item 1 больше не предписывает файл, а ветвится по первому подходящему случаю:

- **(a) энтрипойнт уже есть** (`make check`, `npm test`, `just check`, `tox`) → **он и есть
  оракул**; назвать его в CLAUDE.md и остановиться.
- **(b) проверка — одна известная команда** → задокументировать однострочник, файл **не создавать**.
- **(c) энтрипойнта нет И** проверка многогейтовая (тесты + линт + формат + старт приложения)
  **или** нужна env-подготовка (venv, docker, экспортируемые переменные) → авторить **один**
  скрипт в **`scripts/init.sh`** (или по конвенции проекта `bin/`/`tools/`), **не в корне**.

Только (c) вообще заслуживает файла. Корень оставлен продуктовым — `init.sh` убран из списка
root-файлов, session-start-ритуал переехал целиком в CLAUDE.md.

Рябь по всей поверхности, где `init.sh` был данностью: `SKILL.md` (строка Phase 5),
`operator-playbook.md` §2 и §3, `evidence-executor.md` (шаг Oracle), Phase 7 run-it-once,
`project-docs/workflow.md` шаг 3, `audit-checklist.md` (пост-рефреш прогон оракула).

## Проверка
- **Поведенческий A/B на трёх фикстурах** (headless `claude --print`, чистые git-репозитории,
  машинный критерий — сессия обязана вернуть `ORACLE_COMMAND=` и `CREATED=`). Контроль на старом
  тексте — через `git stash` плагина, то есть старая редакция ехала тем же symlink-догфудом, что
  и новая; сравнивались не формулировки, а то, что сессия оставила на диске. По одному прогону на
  ячейку — это дизайн-решение, а не статистика.
  - **Есть `make check`** — старый текст: `make check`, файла нет. Новый: то же. **Ветка (a)
    дельты не дала** — модель и без явного предписания переиспользовала энтрипойнт. Формулировка
    (a) остаётся страховкой, а не почином измеренного дефекта.
  - **Только `pytest -q`, энтрипойнта нет** — старый текст: **создал `./init.sh` в корне**,
    полезная нагрузка целиком `exec "$PYTEST" -q`, то есть пятнадцать строк bash вокруг одной
    команды. Новый: `ORACLE_COMMAND=pytest -q`, `CREATED=CLAUDE.md` — файла нет. **Это и есть
    измеренный дефект, который правка чинит** (ветка (b)).
  - **Много гейтов, энтрипойнта нет** — новый текст: `scripts/init.sh`, корень чист, скрипт
    реально зелёный (`ORACLE: GREEN`, exit 0), с `cd "$(dirname "$0")/.."` и по-инструментным
    разрешением venv — выученные грабли ветку (c) не потеряли. Старый эквивалент этой фикстуры —
    демо-проект consumer-journey, где тот же класс проекта получил корневой `init.sh`.
- Связность поверхности: `grep -rn "init\.sh" plugins/` — все шесть оставшихся упоминаний имеют
  форму `scripts/init.sh` и условны («only if none exists» / ветка (c)); ссылок на корневой
  `init.sh` не осталось.
- `release.sh` шаг 2 (симуляция): `workflow.md` модифицирован → стамп обязан быть v1.16.1;
  после бампа проверка проходит. Два других project-docs не тронуты — их стампы (v1.9.2,
  v1.10.1) сохранены, провенанс пофайловый.

## Провенанс записи
Двухслойный. Правку сделала одна сессия; CHANGELOG, стамп и первую редакцию этой записи
восстановила по диффу параллельная релизная — «что» и «почему» она вычитала из самого текста, а
эмпирику зафиксировать не могла и честно пометила пробел. Пробел закрыт авторской сессией задним
числом: раздел «Проверка» выше — её замеры, не реконструкция; заодно в «Затронутые файлы» добавлен
`audit-checklist.md`, который реконструкция по диффу пропустила (файл несёт правки двух разных
дефектов сразу, и #12 забрал его целиком).

Провокация правки — не аудит и не эксперимент, а разбор с оператором: на вопрос «почему этот файл
вообще должен лежать в корне» не нашлось ответа лучше, чем «так предписывает чек-лист». Отсюда и
асимметрия результата — ветки (b) и (c) чинят измеренное, ветка (a) добавлена по симметрии.
